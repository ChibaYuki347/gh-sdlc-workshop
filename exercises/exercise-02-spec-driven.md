# 演習2: 仕様駆動開発（SDD）演習

**対応セッション**: セッション3 - AIを活用した要件定義と仕様駆動開発
**所要時間**: 約45分

---

## 🎯 演習のゴール

- ビジネス要件を構造化された仕様書に変換するプロセスを体験する
- Copilot Chat を活用して要件の壁打ちと仕様の精緻化を行う
- ビジネスメンバーとITメンバーが協働して仕様を作成する

---

## 📋 シナリオ

> **あなたのチームは保険CRMシステムの開発チームです。**
> 営業部門から以下の要望が届きました:
>
> 「保険契約の更新日が近づいている顧客を自動でリストアップして、営業担当者に通知できるようにしてほしい。更新漏れによる契約失効を防ぎたい。」

---

## 📝 演習手順

### Phase 1: Copilot Chat で要件を整理する（10分）

**全員で実施**

VS Code で Copilot Chat を開き、以下のプロンプトを入力してください:

```
あなたは保険CRMシステムの要件定義を手伝うアシスタントです。

営業部門から「契約更新通知機能」の要望が来ています。
以下の情報を元に、この機能の要件を整理してください。

【背景】
- 保険契約には更新日（endDate）がある
- 更新日を過ぎると契約が失効する
- 現在は営業担当者が手動で更新日を確認している
- 更新漏れによる失効が月に数件発生している

【既存システム情報】
- Java/Spring Boot のWebアプリケーション
- 顧客(Customer)と保険契約(Policy)のテーブルがある
- REST API として提供

以下を整理してください:
1. 機能概要
2. ユーザーストーリー
3. 機能要件（箇条書き）
4. 非機能要件
5. 考慮すべきエッジケース
```

> 💡 Copilot の回答を読んで、チームで「これで十分か？何が足りないか？」を議論してください

### Phase 2: 仕様書を作成する（20分）

**役割分担して実施**

[仕様書テンプレート](../specs/sample-spec.md) を参考に、チームで仕様書を作成してください。

```bash
# 仕様書ファイルを作成
cp specs/sample-spec.md specs/renewal-notification-spec.md
```

#### ビジネスメンバーの担当部分:

```markdown
## 1. 概要
（この機能が何のために必要か、ビジネス価値は何か）

## 2. ユーザーストーリー
（誰が、何のために、何をしたいか）

## 7. 受け入れ条件
（この機能が完成したと言える条件）
```

#### ITメンバーの担当部分:

```markdown
## 3. 機能要件
（具体的な機能仕様）

## 4. API設計
（エンドポイント、リクエスト/レスポンス）

## 5. データモデル
（テーブル構造、関連）

## 6. エッジケース
（異常系、境界値）
```

### Phase 3: Copilot Chat で仕様をレビューする（10分）

作成した仕様書を Copilot Chat に貼り付けてレビューを依頼:

```
以下の仕様書をレビューしてください。
抜け漏れ、矛盾、改善すべき点を指摘してください。
特にエッジケースと受け入れ条件の網羅性を重点的に確認してください。

---
（作成した仕様書の内容をここに貼り付け）
---
```

レビュー結果を元に仕様書を修正してください。

### Phase 4: タスク分解（5分）

```
この仕様書を実装するためのタスクをGitHub Issue形式で分解してください。
各タスクには以下を含めてください:
- タイトル（feat: / refactor: / test: のプレフィックス付き）
- 説明（2-3行）
- 変更対象ファイル
- 依存関係（先に完了すべきタスクがあれば）
```

生成されたタスクを GitHub Issues に登録してください。

## 💬 ディスカッションポイント

演習後、チーム内で以下を議論してください:

1. Copilot との壁打ちで気づいた「自分たちだけでは見落としていた点」は？
2. ビジネスメンバーとITメンバーの役割分担は適切だったか？
3. この仕様書があれば、実装の手戻りはどの程度防げそうか？
