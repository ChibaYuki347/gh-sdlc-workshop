## 📖 アジェンダ

### 1. 仕様駆動開発（SDD）と Spec Kit 入門（15分）

#### 従来の開発 vs 仕様駆動開発

```
【従来】
  ビジネス要件（口頭/Excel）→ SE が解釈 → 設計書 → 実装
  └ 要件の意図が伝言ゲームで劣化

【仕様駆動開発（SDD）with Spec Kit】
  ビジネス要件 → Specify（仕様化）→ Plan（技術計画）→ Tasks（タスク分解）→ Implement（実装）
  └ 仕様書がそのまま実行可能なコードの源泉に
  └ 各フェーズの成果物がリポジトリで管理される
```

#### Spec Kit とは

[GitHub Spec Kit](https://github.com/github/spec-kit) は、仕様駆動開発を実践するためのオープンソースツールキットです。

**Spec Kit の 4 フェーズ**:

| フェーズ | コマンド | 成果物 | 説明 |
|---------|---------|--------|------|
| **Specify** | `/speckit.specify` | 仕様書 | What/Why を定義。ユーザーストーリー、要件、ユースケースを記述 |
| **Plan** | `/speckit.plan` | 技術計画 | アーキテクチャ、技術選定、設計方針を決定 |
| **Tasks** | `/speckit.tasks` | タスク一覧 | 仕様と計画をもとに実装タスクを分解 |
| **Implement** | Copilot Agent | コード | タスクに基づいてAIがコードを生成 |

#### Spec Kit のディレクトリ構成

```
.specify/              ← Spec Kit のワークスペース
├── spec.md            ← 仕様書（Specify フェーズの成果物）
├── plan.md            ← 技術計画（Plan フェーズの成果物）
└── tasks.md           ← タスク一覧（Tasks フェーズの成果物）
.github/
├── prompts/           ← Copilot Chat スラッシュコマンド
└── copilot-instructions.md  ← プロジェクトのルール・コンテキスト
```

#### 良い仕様書の条件

1. **具体的**: 「検索機能を改善する」ではなく「姓・名・メールアドレスで部分一致検索できる」
2. **テスト可能**: 受け入れ条件が明確
3. **構造化**: 入出力、制約、エッジケースが整理されている

### 2. Spec Kit の初期セットアップとコード理解（15分）

#### Spec Kit の初期化

VS Code のターミナルで以下を実行:

```bash
# Spec Kit の初期化（プロジェクトに .specify/ と .github/prompts/ を生成）
npx @github/specify init
```

> 💡 初期化により、`.specify/` ディレクトリと `.github/prompts/` 配下に Spec Kit 用のスラッシュコマンドが追加されます

#### Copilot Chat でコードを理解する

VS Code で `CustomerService.java` を開き、Copilot Chat に以下を質問してください:

```
プロンプト例:
1. 「このクラスの責務を一覧にしてください」
2. 「このクラスの設計上の問題点を指摘してください」
3. 「calculatePremium メソッドのビジネスロジックを日本語で説明してください」
```

#### 発見すべき課題

Copilot Chat との対話を通じて、以下の課題を発見できるはずです:

- **God Class**: CustomerService に複数の責務が集中
- **マジックナンバー**: ポリシーステータスの数値、保険料率のハードコーディング
- **入力バリデーション不足**: null チェック、範囲チェックの欠如
- **テストの不在**: ビジネスロジックのテストがない

### 3. Spec Kit で仕様書を作成する（30分）

#### シナリオ

> **ビジネス要件**: 保険代理店から「顧客の保険契約更新が近づいたときに自動で通知リストを作成する機能がほしい」という要望が来ています。

#### Step 1: Specify フェーズ — 仕様の作成

Copilot Chat で `/speckit.specify` コマンドを使用:

```
/speckit.specify

保険CRMアプリケーションに「契約更新通知機能」を追加したいです。

【背景】
- 保険契約には更新日（endDate）がある
- 更新日を過ぎると契約が失効する
- 現在は営業担当者が手動で更新日を確認している
- 更新漏れによる失効が月に数件発生している

【要望】
- 保険契約の更新日が30日以内に迫っている顧客をリストアップ
- 通知対象リストを返すAPIを作りたい
- 営業担当者別にフィルタできると嬉しい
```

> Spec Kit が `.specify/spec.md` に構造化された仕様書を生成します。
> 生成された仕様書をチームで確認し、ビジネスメンバーが要件の妥当性を検証してください。

**作業分担の例**:
- **ビジネスメンバー**: 概要、ビジネス要件、受け入れ条件をレビュー・修正
- **ITメンバー**: 技術仕様、API設計、エッジケースをレビュー・追記

#### Step 2: Plan フェーズ — 技術計画の立案

仕様書が確定したら、技術計画を作成します:

```
/speckit.plan
```

> Spec Kit が `.specify/plan.md` に技術計画を生成します。
> アーキテクチャ、使用技術、データモデルの設計方針が含まれます。

チームで計画を確認し、必要に応じて修正してください:

```
確認ポイント:
□ 既存の Spring Boot アーキテクチャとの整合性
□ Customer / Policy モデルとの関連
□ パフォーマンスへの考慮
□ セキュリティへの考慮
```

#### Step 3: Tasks フェーズ — タスク分解

技術計画が確定したら、実装タスクに分解します:

```
/speckit.tasks
```

> Spec Kit が `.specify/tasks.md` にタスク一覧を生成します。
> 各タスクには説明、対象ファイル、依存関係が含まれます。

生成されたタスクを確認し、GitHub Issues として登録してください。

### 4. 仕様書のレビューと改善（15分）

#### Copilot Chat で仕様をレビューする

```
プロンプト:
「.specify/spec.md の仕様書をレビューしてください。
抜け漏れ、矛盾、改善点を指摘してください。
特にエッジケースと受け入れ条件の網羅性を重点的に確認してください。」
```

レビュー結果を元に仕様書を修正し、コミットしてください:

```bash
git add .specify/
git commit -m "docs: 契約更新通知機能の仕様・計画・タスクを追加"
```

> 💡 **ポイント**: Spec Kit の成果物はすべてリポジトリで管理されます。
> 仕様の変更履歴が Git で追跡でき、「なぜこの仕様になったか」が後から確認できます。

## 💡 講師向けメモ

- Spec Kit の初期化がうまくいかない場合は、手動で `.specify/` ディレクトリを作成して進めても可
- ビジネスメンバーの積極的な参加を促す。仕様の「What」を定義するのはビジネスの役割
- `/speckit.specify` の出力は毎回異なる。参加者ごとに結果が違うことを事前に説明
- 仕様書の完成度より、「SDDの4フェーズのプロセス」を体験することが重要
- 各チームが作成した仕様書を簡単に共有する時間を設けると良い
