# 演習4: 仕様駆動での機能追加演習

**対応セッション**: セッション4 - 仕様駆動での機能追加
**所要時間**: 約40分

---

## 🎯 演習のゴール

- 経営課題を起点に、チームで機能を企画する
- 仕様駆動開発（SDD）のフローを自律的に回す
- ビジネスメンバーとITメンバーが協働して動くコードを作る

---

## 📋 経営課題シート

以下から **チームで1つ選んで** ください:

| # | 経営課題 | ヒント |
|---|---------|--------|
| A | 契約の解約率が高い | 更新期限の管理、顧客フォロー |
| B | 営業の生産性が低い | 検索の高度化、ダッシュボード強化 |
| C | コンプライアンスリスク | 変更履歴、監査ログ、遷移ルール |
| D | データ活用の遅れ | 集計レポート、CSV出力、可視化 |
| E | 自由テーマ | チーム独自の課題を設定 |

---

## 📝 演習手順

### Phase 1: 機能企画（10分）

#### Step 1: 課題を選び、解決する機能を決める

チーム全員で話し合い、以下を決定してください:

- **選んだ経営課題**: （A〜E）
- **実装する機能名**: （例: 契約更新リマインダー）
- **ユーザーストーリー**:

```
[誰]として、[何]がしたい。なぜなら[理由]だから。

例:
営業担当者として、更新期限が近い契約の一覧を見たい。
なぜなら、期限前に顧客フォローを行い、解約を防ぎたいから。
```

#### Step 2: スコープを絞る

90分で実装可能な範囲に絞りましょう:

| 含める（Must） | 含めない（Nice to have） |
|---------------|----------------------|
| API 1〜2本 | 複雑な認証・認可 |
| 画面 1〜2ページ | 外部API連携 |
| 基本的なCRUD | メール送信機能 |

### Phase 2: 仕様書の作成（15分）

#### Step 1: Copilot で仕様書を生成

```
プロンプト:
「以下の機能について、仕様書を作成してください。

## 経営課題
[選択した課題の説明]

## 機能概要
[チームで決めた機能の説明]

## ユーザーストーリー
[Step 1 で記述したユーザーストーリー]

以下の構成で出力してください:
1. 機能要件
2. 画面設計（主要な画面の説明）
3. API設計（エンドポイント・メソッド・パラメータ）
4. データモデル（必要なエンティティ・フィールド）
5. 受け入れ条件」
```

#### Step 2: ビジネスメンバーによるレビュー

> 🔍 **ビジネスメンバー**: 以下をチェックしてください
> - 機能要件が経営課題の解決につながっているか？
> - ユーザーストーリーの意図が正しく反映されているか？
> - 受け入れ条件に漏れがないか？

指摘があれば Copilot に修正を依頼します。

#### Step 3: 仕様書を保存

```bash
# specs/ ディレクトリに保存
mkdir -p specs/team-X
# Copilot の出力を specs/team-X/feature-spec.md にコピー
```

### Phase 3: 実装（15分）

#### Step 1: 実装計画を立てる

```
プロンプト:
「以下の仕様に基づいて、実装計画を立ててください。

[仕様書の内容を貼り付け]

既存のコード構成:
- Controller: app/src/main/java/com/example/crm/controller/
- Service: app/src/main/java/com/example/crm/service/
- Model: app/src/main/java/com/example/crm/model/
- Repository: app/src/main/java/com/example/crm/repository/
- Templates: app/src/main/resources/templates/

まず計画を提示し、承認を得てから実装してください。」
```

> ⚠️ **計画を確認してから「承認します、実装してください」と伝える** — いきなりコードを生成させない

#### Step 2: Agent Mode で段階的に実装

```
プロンプト例:
「計画に沿って実装を開始してください。
まず Step 1（モデル・リポジトリ）から始めてください。」
```

途中で方向性がずれたら、その場で軌道修正できるのが Agent Mode の強みです。

#### Step 3: コンパイル・動作確認

```bash
cd app
mvn compile
mvn spring-boot:run
```

ブラウザで http://localhost:8080/ にアクセスし、追加した機能を確認します。

> 🎉 **全員で画面を見ながら確認** — 「仕様通りか？」をビジネスメンバーが判断

---

## ✅ 完了条件

- [ ] 経営課題を選び、機能を企画している
- [ ] 仕様書（specs/team-X/feature-spec.md）が作成されている
- [ ] ビジネスメンバーが仕様をレビュー・承認している
- [ ] 実装計画を立て、承認してから実装している
- [ ] `mvn compile` が成功する
- [ ] 機能が画面上で動作確認できる
- [ ] 変更がコミットされている

## 💡 ヒント

### 実装が間に合わない場合

スコープを縮小しましょう:
- 画面なしでAPI だけ実装 → curl で動作確認
- 既存画面の一部を修正するだけでも OK

### チームの議論が進まない場合

「A. 契約の解約率が高い → 契約更新リマインダー」を推奨テーマとして採用してください。

### プロンプトのコツ

| パターン | 良い例 | 悪い例 |
|---------|-------|-------|
| **具体的** | 「更新期限が30日以内の契約一覧APIを作って」 | 「機能を作って」 |
| **段階的** | 「まずモデルを作り、次にリポジトリ」 | 「全部一度に作って」 |
| **制約の明示** | 「既存のCustomerエンティティに関連づけて」 | 「新しいのを作って」 |

> 💡 **振り返り**: 午前中のガイド付きSDDと、午後の自律的SDDの違いを感じてみてください。
> 仕様の質がコードの質を決める — これが仕様駆動開発の本質です。
