## 📖 アジェンダ

### 1. 2つのエージェントの使い分け（10分）

#### Coding Agent vs Agent Mode

```
┌─────────────────────────────────┬──────────────────────────────────┐
│    Copilot Coding Agent         │    Copilot Agent Mode            │
│    （クラウド・自律型）           │    （IDE・対話型）                │
├─────────────────────────────────┼──────────────────────────────────┤
│ ✅ Issue をアサインすると自動実行 │ ✅ IDE内で対話しながら実装        │
│ ✅ ブランチ・PR も自動作成       │ ✅ リアルタイムで方向性を指示可能  │
│ ✅ 明確な仕様がある新機能向き    │ ✅ 探索的・段階的な作業向き       │
│ ❌ 実行中の介入は難しい          │ ❌ 手動でコミット・PR作成が必要   │
├─────────────────────────────────┼──────────────────────────────────┤
│ 午前中（セッション3）で体験済み   │ このセッションで体験              │
└─────────────────────────────────┴──────────────────────────────────┘
```

**使い分けの指針**:
- **Coding Agent**: 仕様が明確 → Issue に書いて任せる
- **Agent Mode**: 試行錯誤が必要、既存コードの理解・改善 → 対話しながら進める

### 2. Copilot Chat でレガシーコードを理解する（20分）

#### 全員で実施

VS Code で `CustomerService.java` を開き、Copilot Chat に質問:

```
プロンプト:
「CustomerService.java を分析して、以下を整理してください:
1. このクラスが持つ全ての責務の一覧
2. 各メソッドの設計上の問題点
3. リファクタリングの優先順位」
```

#### チームでの役割

- **ビジネスメンバー**: Copilot の分析結果を読み、ビジネスロジック（保険料計算等）が正しいか確認
- **ITメンバー**: 設計上の問題点を理解し、リファクタリング方針を検討

#### 発見すべき課題

| 問題 | 説明 | リスク |
|------|------|--------|
| God Class | 顧客管理・保険料計算・レポート・通知が1クラスに | 変更時の影響範囲が予測困難 |
| マジックナンバー | `policyStatus == 1` 等のハードコーディング | 意味が不明、バグの温床 |
| バリデーション不足 | null チェック、入力範囲チェックの欠如 | 本番障害のリスク |
| テスト不在 | ユニットテストがない | リファクタリングの安全性が担保できない |

### 3. Agent Mode でリファクタリング（40分）

#### Phase 1: 保険料計算の分離（20分）

VS Code の Copilot Chat で Agent Mode を使用:

```
プロンプト:
「calculatePremium メソッドを以下の方針でリファクタリングしてください:

1. PremiumCalculatorService を新規作成
   - 場所: app/src/main/java/com/example/crm/service/PremiumCalculatorService.java

2. マジックナンバーを定数に置き換え
   - 基本保険料、年齢係数、喫煙者割増率を定数化
   - 消費税率を定数化

3. 各保険タイプの計算を個別メソッドに分離

4. 入力バリデーション追加
   - policyType が null または未知の場合は IllegalArgumentException
   - age が 0未満または150超の場合は IllegalArgumentException

5. CustomerService の calculatePremium は新サービスに委譲

既存のAPIの振る舞いは変更しないでください。」
```

**ビジネスメンバーの確認ポイント**:
- リファクタリング前後で保険料の計算結果が変わっていないか
- 定数化された料率がビジネスルールと一致しているか

#### Phase 2: ステータス管理の改善（20分）

```
プロンプト:
「policyStatus のマジックナンバー（0=申請中, 1=有効, 2=失効, 3=解約）を改善:

1. PolicyStatus enum を作成
   - 各ステータスに日本語ラベルと数値コードを持たせる

2. ステータス遷移のバリデーション追加
   - 有効な遷移: 申請中→有効、有効→失効/解約
   - 無効な遷移: 失効→有効（不可）、解約→いずれにも変更不可

3. Customer エンティティの policyStatus を enum に変更」
```

#### コンパイル確認

```bash
cd app
mvn compile
```

### 4. Agent Mode の効果的な使い方（10分）

#### プロンプトのコツ

| パターン | 良い例 | 悪い例 |
|---------|-------|-------|
| **具体的な指示** | 「CustomerServiceのsearchCustomersをJPA Specificationで書き直して」 | 「検索を改善して」 |
| **コンテキスト** | 「Spring Boot 3.2 + JPA環境で、既存のRepositoryを拡張して」 | 「APIを作って」 |
| **段階的** | 「まずインターフェースを定義し、次に実装を作成して」 | 「全部作って」 |
| **制約の明示** | 「既存APIの互換性を保ちながら内部実装を変更」 | 「リファクタリングして」 |

### 5. コミットと午前の Coding Agent PR の確認（10分）

```bash
git add .
git commit -m "refactor: 保険料計算の分離とステータス管理の改善"
git push
```

## 💡 講師向けメモ

- リファクタリングは全てを完了する必要はない。プロセスの体験が目的
- ビジネスメンバーには「リファクタリング前後のコードの可読性の違い」に注目してもらう
- ビジネスメンバーの役割を明示する（計算結果の検証、仕様の正しさの確認）
- 午前の Coding Agent PR が完成していれば、ここで確認する時間を設ける
