# セッション4: 仕様駆動での機能追加 — ビジネスとITの協働実装

**時間**: 13:00 - 14:30（90分）

---

## 🎯 セッションのゴール

- 経営課題を起点に、チームで独自の機能を企画・仕様化・実装する体験をする
- **ビジネスメンバー**（仕様策定・レビュー）と **ITメンバー**（AI活用実装）の協働を実践する
- Copilot Agent Mode を使った対話的な開発で、仕様から動くコードへの変換を体験する
- 仕様駆動開発の2サイクル目として、午前中に学んだSDDフローを自律的に回す

---

## 📖 アジェンダ

### 1. 午前の振り返りと午後の進め方（10分）

#### 午前中に体験したSDD フロー

```
午前（セッション3）:
  仕様を書く → 計画を立てる → タスクに分解 → Issue 作成 → 実装
  └ ガイド付きで「契約更新通知機能」をテーマに体験

午後（このセッション）:
  経営課題から機能を企画 → 仕様策定 → 実装 → 動作確認
  └ チーム独自のテーマで自律的にSDDサイクルを回す
```

#### 午前の Coding Agent PR の確認（該当する場合）

> ⏰ 午前中に Coding Agent にアサインした Issue がある場合は、PR の状態を確認しましょう。
>
> **Coding Agent が利用できない環境の場合**: この確認はスキップし、午後の機能追加に進みます。

### 2. 経営課題のブリーフィング（10分）

#### 📋 シナリオ: CRMシステムの経営課題

あなたのチームは保険CRMシステムの改善プロジェクトに参加しています。
経営層から以下の課題が共有されました。**チームで1つ選び、解決する機能を企画してください。**

| # | 経営課題 | 背景 | 期待する効果 |
|---|---------|------|-------------|
| A | **契約の解約率が高い** | 有効契約の解約が増加。契約更新時期の前に顧客フォローができていない | 解約率の低減、顧客満足度の向上 |
| B | **営業の生産性が低い** | 顧客情報の検索・閲覧に時間がかかる。重要顧客の優先対応ができていない | 営業プロセスの効率化 |
| C | **コンプライアンスリスク** | 契約ステータスの変更履歴が残らない。監査対応が手作業 | 監査対応コスト削減、リスク低減 |
| D | **データ活用の遅れ** | 契約データの分析・可視化ができていない。経営判断がKKD（勘・経験・度胸）頼み | データドリブンな意思決定の実現 |
| E | **自由テーマ** | チーム独自の課題を設定してOK | — |

> 💡 **ビジネスメンバーの出番**: どの課題が自社で最もリアリティがあるか、業務知識を活かして選定してください。

#### 各課題の機能アイデア例

<details>
<summary>A. 契約の解約率が高い → 機能アイデア例（クリックで展開）</summary>

- **契約更新リマインダー**: 更新期限の30日前に一覧表示する画面
- **顧客スコアリング**: 解約リスクの高い顧客を自動判定
- **フォロー履歴管理**: 顧客へのフォローアクション記録機能

</details>

<details>
<summary>B. 営業の生産性が低い → 機能アイデア例（クリックで展開）</summary>

- **高度な顧客検索**: 複数条件での絞り込み検索（契約種別、地域、金額範囲）
- **ダッシュボード強化**: 重要KPIの一覧表示（本日期限、高額契約、未対応顧客）
- **顧客タグ機能**: VIP、要フォローなどのラベル管理

</details>

<details>
<summary>C. コンプライアンスリスク → 機能アイデア例（クリックで展開）</summary>

- **ステータス変更履歴**: 契約ステータスの変更日時・変更者・理由の記録
- **監査ログ画面**: 変更履歴の検索・エクスポート機能
- **ステータス遷移ルール**: 不正な遷移（失効→有効等）の防止

</details>

<details>
<summary>D. データ活用の遅れ → 機能アイデア例（クリックで展開）</summary>

- **契約分析レポート**: 種別ごとの契約数・保険料の集計・グラフ表示
- **月次トレンド表示**: 新規契約・解約の推移表示
- **CSV エクスポート**: データ分析ツール連携用のデータ出力

</details>

### 3. チームで機能を企画・仕様化する（25分）

#### ビジネスとITの役割分担

```
┌─────────────────────────────┬──────────────────────────────────┐
│    ビジネスメンバーの役割       │    ITメンバーの役割                │
├─────────────────────────────┼──────────────────────────────────┤
│ ✅ 経営課題から機能を選定      │ ✅ 技術的な実現可能性を検討        │
│ ✅ ユーザーストーリーを記述     │ ✅ Copilot で実装計画を立案        │
│ ✅ 受け入れ条件を定義          │ ✅ タスク分解と工数の見積もり       │
│ ✅ 仕様の最終レビュー・承認     │ ✅ 仕様に基づく実装の実行          │
└─────────────────────────────┴──────────────────────────────────┘
```

#### Step 1: 機能の企画（10分）

チーム全員で経営課題を選び、解決する機能を決定します:

1. **経営課題の選定**: 上の表から1つ選ぶ（または自由テーマ）
2. **機能のスコープ**: 90分以内で実装可能な範囲に絞る
3. **ユーザーストーリー**: 「〜として、〜したい。なぜなら〜だから」の形式で記述

#### Step 2: 仕様書の作成（15分）

午前中に学んだ SDD フローを使って仕様を作成します:

**方法A: Prompt Files を使う場合**
```
/specify を使って、以下の機能の仕様書を作成してください:
[チームが決めた機能の説明]
```

**方法B: Copilot Chat で直接作成する場合**
```
プロンプト:
「以下の機能について、仕様書を作成してください。

## 経営課題
[選択した課題]

## 機能概要
[チームで決めた機能]

## ユーザーストーリー
[記述したユーザーストーリー]

以下の構成で仕様書を出力してください:
1. 機能要件（必須）
2. 画面設計（主要な画面の説明）
3. API設計（エンドポイントの定義）
4. 受け入れ条件（テスト観点）」
```

> 💡 **ビジネスメンバー**: 仕様書の機能要件とユーザーストーリーを必ずレビューしてください。
> 「この仕様で業務課題は解決するか？」が最も重要な判断ポイントです。

### 4. Copilot Agent Mode で実装する（35分）

#### Step 1: 実装計画の策定

```
プロンプト:
「以下の仕様に基づいて、既存のCRMアプリケーションへの実装計画を立ててください。

[作成した仕様書を貼り付け]

既存のコード構成:
- Controller: app/src/main/java/com/example/crm/controller/
- Service: app/src/main/java/com/example/crm/service/
- Model: app/src/main/java/com/example/crm/model/
- Repository: app/src/main/java/com/example/crm/repository/
- Templates: app/src/main/resources/templates/

まず実装計画を提示してください。承認後に実装を開始します。」
```

> 💡 **重要**: 計画を確認してから実装に進む。AIに「いきなりコードを書かせない」のが仕様駆動開発のポイントです。

#### Step 2: 段階的な実装

計画を承認したら、Agent Mode で段階的に実装を進めます:

```
プロンプト例:
「計画の Step 1 を実装してください。
[モデル/リポジトリ/サービス/コントローラーの順で段階的に]」
```

**ビジネスメンバーの確認ポイント**:
- 画面のレイアウトや表示項目がユーザーストーリーに合っているか
- エラーメッセージがエンドユーザーにわかりやすいか
- 受け入れ条件を満たしているか

#### Step 3: 動作確認

```bash
cd app
mvn spring-boot:run
```

ブラウザで http://localhost:8080/ にアクセスし、追加した機能が動作することを確認します。

> 🎉 **ビジネスメンバーとITメンバーが一緒に画面を見ながら確認** — これが仕様駆動開発の醍醐味です。

### 5. コミット・PR 作成（10分）

```bash
git add .
git commit -m "feat: [チームの機能名]の実装"
git push origin feature/team-X-[機能名]
```

GitHub で Pull Request を作成:

```
プロンプト（PR 説明文の生成）:
「このブランチの変更内容に基づいて、Pull Request の説明文を日本語で作成してください。
経営課題、実装した機能、変更内容、テスト方法を含めてください。」
```

## 💡 講師向けメモ

- **最大の見せ場は「チームが自分たちで課題を選び、仕様を書き、動くコードにする」プロセス**
- 経営課題の選定で迷うチームには「A. 解約率」を推奨（仕様がイメージしやすい）
- ビジネスメンバーには仕様レビューの際「これで業務は回るか？」と問いかける
- 実装が間に合わないチームには、最小限の機能（API 1本 + 画面1つ）に絞るようガイド
- 「仕様 → 計画 → 承認 → 実装」のステップを省略させない（計画なしでコード生成は NG）
- 各チームの進捗が異なるため、早く終わったチームには機能拡張や2つ目の課題に挑戦を促す
