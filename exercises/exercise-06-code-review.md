# 演習6: コードレビュー演習

**対応セッション**: セッション5 - テスト生成とセキュアなコードレビュー
**所要時間**: 約30分

---

## 🎯 演習のゴール

- Copilot を活用して Pull Request の説明文を作成する
- チーム間でコードレビューを実施する
- セキュリティ観点でのレビューを体験する

---

## 📝 演習手順

### Step 1: Pull Request を作成する（10分）

#### 変更をコミット・プッシュ

```bash
# すべての変更をステージング
git add .

# コミットメッセージを作成（Copilotに提案してもらう）
git commit -m "feat: 契約更新通知機能の追加とレガシーコードのリファクタリング

- 契約更新通知サービスの新規実装
- 保険料計算ロジックの分離とリファクタリング
- PolicyStatus enum によるステータス管理の改善
- ユニットテストの追加"

# プッシュ
git push origin feature/team-X-renewal-notification
```

#### PR説明文を Copilot で生成

VS Code Copilot Chat で:

```
このブランチの変更内容に基づいて、Pull Requestの説明文を日本語で作成してください。

以下のセクションを含めてください:
## 概要
## 変更内容
## テスト結果
## レビューポイント（レビュアーに特に見てほしい点）
## チェックリスト
```

GitHub で PR を作成し、生成された説明文を使用してください。

### Step 2: セキュリティレビュー（10分）

自チームのコードに対して、Copilot でセキュリティレビューを実施:

```
以下のコントローラクラスのセキュリティ上の問題点を
OWASP Top 10 の観点から分析してください:

（CustomerController.java の内容を貼り付け）

各問題について:
1. リスクレベル（高/中/低）
2. 問題の説明
3. 修正方法
を示してください。
```

**発見すべき典型的な問題**:

| 問題 | リスク | 説明 |
|------|--------|------|
| 入力バリデーション不足 | 高 | SQLインジェクション、XSSの可能性 |
| エラー情報の過剰公開 | 中 | スタックトレースがそのまま返される |
| 認証・認可の欠如 | 高 | 誰でもAPIにアクセスできる |
| レート制限なし | 中 | DoS攻撃に脆弱 |

### Step 3: チーム間コードレビュー（10分）

**他チームのPRをレビューする**

講師がチーム間のPR割り当てを指示します:
- チーム1 → チーム2のPRをレビュー
- チーム2 → チーム3のPRをレビュー
- （以下繰り返し）

#### レビュー観点

```
□ コードの可読性
  - 変数名・メソッド名は意味が分かるか
  - コメントは適切か

□ 仕様との整合性
  - 仕様書の要件を満たしているか
  - 受け入れ条件がテストで確認されているか

□ 設計・構造
  - 適切な責務分離がされているか
  - 既存コードとの一貫性があるか

□ エラーハンドリング
  - null チェックは適切か
  - 例外処理は適切か

□ テスト
  - テストカバレッジは十分か
  - エッジケースがテストされているか
```

#### Copilot を使ったレビュー支援

レビュー対象のコードを選択して:

```
このコードをレビューしてください。
以下の観点で問題点や改善提案があれば教えてください:
1. バグの可能性
2. パフォーマンスの懸念
3. セキュリティリスク
4. 可読性の改善
```

#### レビューコメントの書き方

GitHub の PR 画面で、コードの該当行にコメントを追加:

```markdown
<!-- 質問の場合 -->
❓ この条件分岐で endDate が null の場合はどうなりますか？

<!-- 提案の場合 -->
💡 ここは Optional.ofNullable() を使うとより安全になると思います。

<!-- 必須修正の場合 -->
🔴 入力バリデーションがないため、不正な値でも処理が通ってしまいます。
@Valid アノテーションの追加を検討してください。

<!-- 称賛の場合 -->
👍 マジックナンバーを enum に置き換えているのが良いですね！
```
