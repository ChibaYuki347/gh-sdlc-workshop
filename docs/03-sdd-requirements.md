# セッション3: AIを活用した要件定義と仕様駆動開発（SDD）の体験

**時間**: 10:45 - 12:00（75分）

---

## 🎯 セッションのゴール

- ビジネス要件を仕様書（Markdown）として記述する方法を学ぶ
- Copilot Chat を使った要件の壁打ちと技術計画の立案を体験する
- 仕様駆動開発（SDD）のワークフローを理解する

---

## 📖 アジェンダ

### 1. 仕様駆動開発（SDD）入門（15分）

#### 従来の開発 vs 仕様駆動開発

```
【従来】
  ビジネス要件（口頭/Excel）→ SE が解釈 → 設計書 → 実装
  └ 要件の意図が伝言ゲームで劣化

【仕様駆動開発】
  ビジネス要件（構造化 Markdown）→ AI が解釈 → 実装コード
  └ 仕様書がそのまま実行可能なコードの源泉に
```

#### 良い仕様書の条件

1. **具体的**: 「検索機能を改善する」ではなく「姓・名・メールアドレスで部分一致検索できる」
2. **テスト可能**: 受け入れ条件が明確
3. **構造化**: 入出力、制約、エッジケースが整理されている

### 2. 現状のCRMアプリを理解する（15分）

#### Copilot Chat でコードを理解する

VS Code で `CustomerService.java` を開き、Copilot Chat に以下を質問してください:

```
プロンプト例:
1. 「このクラスの責務を一覧にしてください」
2. 「このクラスの設計上の問題点を指摘してください」
3. 「calculatePremium メソッドのビジネスロジックを日本語で説明してください」
```

#### 発見すべき課題

Copilot Chat との対話を通じて、以下の課題を発見できるはずです:

- **God Class**: CustomerService に複数の責務が集中
- **マジックナンバー**: ポリシーステータスの数値、保険料率のハードコーディング
- **入力バリデーション不足**: null チェック、範囲チェックの欠如
- **テストの不在**: ビジネスロジックのテストがない

### 3. 新機能の仕様書を作成する（30分）

#### シナリオ

> **ビジネス要件**: 保険代理店から「顧客の保険契約更新が近づいたときに自動で通知リストを作成する機能がほしい」という要望が来ています。

#### Step 1: Copilot Chat で要件を壁打ちする

```
プロンプト:
「保険CRMアプリケーションに『契約更新通知機能』を追加したいです。
保険契約の更新日が30日以内に迫っている顧客をリストアップし、
通知対象リストを返すAPIを作りたいです。
この機能の要件を整理するのを手伝ってください。
考慮すべきエッジケースも含めてください。」
```

#### Step 2: 仕様書テンプレートに記入する

[サンプル仕様書テンプレート](../specs/sample-spec.md) を参考に、チームで仕様書を作成してください。

**作業分担の例**:
- **ビジネスメンバー**: 概要、ビジネス要件、受け入れ条件を記述
- **ITメンバー**: 技術仕様、API設計、エッジケースを記述

```markdown
# 契約更新通知機能 仕様書

## 概要
（ビジネスメンバーが記述）

## ビジネス要件
（ビジネスメンバーが記述）

## 技術仕様
### API設計
（ITメンバーが記述）

### データモデル
（ITメンバーが記述）

## 受け入れ条件
（チームで議論して記述）

## エッジケース
（Copilotの提案を参考にチームで記述）
```

#### Step 3: Copilot Chat で仕様をレビューする

```
プロンプト:
「以下の仕様書をレビューしてください。
抜け漏れ、矛盾、改善点を指摘してください。

（作成した仕様書を貼り付け）」
```

### 4. タスク分解の自動化（15分）

作成した仕様書を元に、Copilot Chat で実装タスクを分解します。

```
プロンプト:
「この仕様書の実装に必要なタスクをGitHub Issue形式で分解してください。
各タスクには以下を含めてください:
- タイトル
- 説明
- 受け入れ条件
- 想定される変更ファイル」
```

生成されたタスクを GitHub Issues として登録してください。

## 💡 講師向けメモ

- ビジネスメンバーの積極的な参加を促す。仕様の「What」を定義するのはビジネスの役割
- Copilot Chat への質問例を画面に表示しておくと参加者が迷わない
- 仕様書の完成度より、「AIと対話して仕様を磨くプロセス」を体験することが重要
- 各チームが作成した仕様書を簡単に共有する時間を設けると良い
